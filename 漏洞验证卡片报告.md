## 前置工作：验真基线

### crash_analyze.md 已纠正的误报
- **GetBinaryProperty / GetStringProperty / GetMultiSzProperty 的 MoreData 模式是安全的**：`ref propertyValueSize` 在第一次调用后被 native 更新，第二次调用前正确 realloc。不存在托管层堆溢出。
- **RCE 不在托管代码**：RCE 必须通过 FrameworkSupport.dll 或 clusapi.dll 的 native 代码。
- **反射型信息泄露不适用于 RPC 层字符串**：所有 RPC `[string]` 参数由 rpcrt4 强制 null 结尾。泄露路径仅存在于 control code 的 raw byte blob 内嵌字符串。

### harness fuzzing 结论
- CreatePropList 经过 4核×2天 fuzzing 无崩溃 → FrameworkSupport.dll 的 property list 解析器健壮性较高。
- **但注意**: fuzzing 仅覆盖 CreatePropList+iteration。PropertyData 构造函数的 wcslen 越界读不会必现崩溃（读到堆内合法内存时静默通过），fuzzing 无法有效检测 InfoLeak 类漏洞。

### OOB Read → 反射型信息泄露 升级分析

参照 TsVmHostWmi.dll 的反射型信息泄露模式（RPC调用1返回脏数据 → 客户端自动将脏数据作为参数发起RPC调用2 → 服务端收网），逐一分析每个 OOB Read 漏洞是否存在"越界数据回传服务端"的闭环。

#### 反射型模式三要素
1. **OOB Read**: 越界读到额外内存数据
2. **数据存储**: 脏数据被存入变量（string/struct）
3. **回传**: 该变量被用于后续 RPC/网络调用，发送回攻击者控制的服务端

#### 逐 VC 分析

| VC | OOB 机制 | 脏数据存储形式 | 是否回传服务端? | 回传路径 | 自动/手动 | 升级结论 |
|----|---------|-------------|--------------|---------|----------|---------|
| VC-02 | `PtrToStringUni` 无界 | `List<string>` | **否** | line 4812: 转 Guid → 本地通知队列 | N/A | **不可升级** |
| VC-03 | PropertyList 值枚举越界 | PropertyData wstring | 同 VC-16 | 同 VC-16 | 同 VC-16 | 归入 VC-16 |
| VC-04 | Pool Drive Info 结构体越界 | 本地 struct 字段 | **否** | 填充 GUI 显示，不回传 | N/A | **不可升级** |
| VC-05 | ParseProperties size 偏大 | PropertyData wstring | 同 VC-16 | 同 VC-16（但实际取 Guid 字段，new Guid() 会截断/异常） | 手动 | 理论归入 VC-16，实际难利用 |
| VC-06 | count 循环越界 PtrToStructure | 本地 struct 字段 | **否** | 填充 SR_RESOURCE_TYPE_REPLICATED_DISK → 本地使用 | N/A | **不可升级** |

#### VC-16 反射型信息泄露的具体回传路径

**路径 A: DnsName → ResetCnoPassword（RPC 回传）**
```
ParseProperties → PropertyData::wcslen越界 → 脏 DnsName string
  → ClusApiAdapter.cs:5378: dnsName = collection["DnsName"].Value.ToString()
  → ClusApiAdapter.cs:5379: NativeMethods.ResetCnoPassword(resourceHandle, dnsName)
    → RPC [in, string] wchar_t* → 发送回集群服务端（即攻击者）
```
- **触发条件**: 用户在 SnapIn 中执行"重置 CNO 密码"操作
- **数据源**: NetworkName 资源的 Private Properties（control code blob）
- **攻击者可见**: ResetCnoPassword 的 RPC 参数中包含越界堆数据

**路径 B: Name → ActiveDirectory.EnableComputerObject（LDAP 回传）**
```
ParseProperties → PropertyData::wcslen越界 → 脏 Name string
  → ClusApiAdapter.cs:5361: cnoNetBiosName = collection["Name"].Value.ToString()
  → ClusApiAdapter.cs:5366: ActiveDirectory.EnableComputerObject(name, cnoNetBiosName, domain)
    → LDAP 调用 → 如攻击者控制 AD 也可见
```
- **触发条件**: 用户执行"修复 VCO"操作
- **回传目标**: Active Directory（非 RPC 服务端），攻击者需同时控制 AD

**路径 C: 任意 String 属性 → SaveProperties（RPC 回传）**
```
ParseProperties → PropertyData::wcslen越界 → 脏属性值
  → 存储在 PResource.Properties / PGroup.Properties
  → 用户在属性面板编辑某属性 → IsModified = true
  → SaveProperties → AddInPropertyList → NativeMethods.AddStringProperty(propList, name, value)
    → ClusterResourceControl(SET_PROPERTIES) → RPC 回传
```
- **触发条件**: 用户编辑并保存被污染的**同一个**属性
- **限制**: IsModified 仅对用户修改的属性为 true；如果用户修改的是属性 A，被污染的属性 B **不会被发送**

#### 与 TsVmHostWmi 反射型的关键差异

| 对比项 | TsVmHostWmi | FailoverClusters (VC-16) |
|-------|------------|-------------------------|
| 自动化程度 | **全自动**: 单函数内 enumerate → loop → call with tainted data | **需手动触发**: 用户必须执行特定操作（重置密码/修复VCO/编辑属性） |
| 触发复杂度 | 发起一次 WMI 查询即可 | 连接集群 + 执行特定管理操作 |
| 回传目标 | 同一 RPC 服务端 | 路径A: 同一 RPC 服务端 ✓; 路径B: AD 服务器（需额外控制） |
| 确定性 | 高（单函数闭环） | 中（依赖用户行为） |
| OOB 机制 | NDR wcslen（固定数组无 null） | PropertyData wcslen（blob 字符串无 null） |

#### 结论

- **VC-16 可升级为"手动反射型信息泄露 (Manual Reflected InfoLeak)"**，路径 A（DnsName → ResetCnoPassword）最强，因为数据直接通过 RPC 回传到攻击者控制的集群服务端。
- **其余 OOB VC (02/04/06/11/12) 不可升级**：越界数据仅用于本地 GUI 显示或类型转换（Guid），不流向任何网络调用。
- **VC-03/05/07 间接归入 VC-16**：它们的 OOB 最终通过 PropertyData wcslen 机制产生脏字符串，是否回传取决于调用方对属性值的使用。
- **全自动反射路径在 FailoverClusters 中未发现**：所有枚举/加载操作中的属性值仅用于本地（GroupType→enum cast, Priority→uint, Type→dictionary lookup, Name→GUI display）。

### FrameworkSupport.dll 逆向分析关键发现（基于反编译源码）

#### PropertyList vs ValueList 安全性差异
| 特性 | PropertyList | ValueList |
|------|-------------|-----------|
| buffer 处理 | **存储原始指针**（不复制） | **复制 buffer** 到 std::vector |
| 构造时验证 | **无 ParseBuffer** — BuildListFromBuffer 仅传递给 lambda | **有 ParseBuffer** — 逐值校验 `offset + aligned(cbLength) + 8 <= size` |
| 迭代边界检查 | `get_item` 仅检查 `bufferEnd - bufferStart < offset + 8`（**部分检查**） | `list_accessor::get` 检查 `vector.size < offset + 8`（**完整检查**） |
| 值枚举循环 | `for (i = ...; *i; i += aligned(i[1]) + 8)` — **无边界检查**，依赖 endmark null DWORD | 通过 get() 返回 null 终止 |

#### PropertyData 构造函数的 wcslen 漏洞（PropertyList.c:515-519）
```c
v8 = v7 + 2;  // 指向字符串数据
v9 = -1i64;
do
    ++v9;
while ( v8[v9] );  // UNBOUNDED wcslen — 忽略 cbLength 字段
std::wstring::assign(this + 24, v8, v9);
```
- **使用 wcslen 而非 cbLength**：即使 CLUSPROP_VALUE.cbLength 指定了大小，PropertyData 仍然用 unbounded scan
- **影响**: 如果 property list blob 中字符串无 null terminator，读到越界内存 → InfoLeak 或 AV
- **但**: blob 需要通过 `BuildListFromBuffer` 解析后才到 PropertyData，如果 endmark 正确放置，字符串通常是 null-terminated

#### list_accessor::get 的 flag_10 旁路（ValueList.c:1130-1131）
```c
if ( !this->flag_10 )
    return (CLUSPROP_VALUE *)(a2 + v4);  // NO bounds check when flag_10 == 0!
```
当 `flag_10 == 0` 时，直接返回指针 **不做边界检查**。需要确认 PropertyList 路径是否设置 flag_10。

### 异常处理分析：DoS/OOB 是否被 try/catch 捕获

对每个 DoS/OOB 漏洞点检查：是否有 try/catch 能捕获该异常类型？如果被捕获且不崩溃，标记为 False Positive。

**关键前提**：
- `ClusterException` 继承自 `Exception`（不是 `SystemException`）。`NullReferenceException`/`FormatException`/`IndexOutOfRangeException` 等 **不是** `ClusterException`。
- .NET 4.0+ 中 `AccessViolationException`（native OOB 读越界触发的 AV）是**不可捕获**的腐败状态异常，任何 catch 块都无法拦截。
- 但 OOB 读通常不触发 AV（读到同堆分配器管理的有效页内），而是静默读到垃圾数据 → **无异常** → 无 DoS，但有数据污染/InfoLeak。

| VC | 异常类型 | 最近 catch | 匹配? | 实际行为 | DoS 结论 |
|----|---------|-----------|------|---------|---------|
| VC-01 | NullReferenceException | `catch(ClusterException)` :838 | **否** | NRE 不是 ClusterException → 传播 | **DoS 成立** |
| VC-02 | AV（如果跨页）或无异常（堆内） | 无 catch | N/A | AV 不可捕获 / 堆内无异常 | **DoS 仅在跨页时成立；堆内读为静默 InfoLeak** |
| VC-03 | native AV 或无异常 | 取决于调用点 | - | 同 VC-02 | **同 VC-02** |
| VC-04 | AV 或无异常 | 无 catch（callback 内） | N/A | 同 VC-02 | **DoS 仅在跨页时；通常静默** |
| VC-05 | NRE/FormatException（if garbage Guid）| **`catch(Exception)` :6648** | **是** | 异常被捕获，log 后 continue | **~~DoS~~ → False Positive（异常被 catch(Exception) 捕获）** |
| VC-06 | AV 或无异常 | **`catch(Exception)` :6599** | AV不可捕获;无异常时无需捕获 | AV仍崩溃;堆内读静默 | **DoS 仅在跨页时（AV 不可捕获）；堆内读静默通过** |
| VC-07 | AV 或无异常（callback 内） | try-finally，无 catch | N/A | AV 传播 | **DoS 仅在跨页时** |
| VC-08 | OutOfMemoryException | try-finally，无 catch | **否** | OOM 传播 | **DoS 成立** |
| VC-10 | ManagementException | 取决于调用者 | 部分 | - | **DoS 成立（部分路径）** |
| VC-11 | AV | 通知循环 catch(ClusterException/Win32Exception) | **否**（AV 不匹配） | AV 不可捕获 → 线程崩溃 | **DoS 仅在跨页时** |
| VC-12 | AV | 同 VC-11 | **否** | 同 VC-11 | **DoS 仅在跨页时** |
| VC-13 | ArgumentOutOfRangeException | try-finally，无 catch | **否** | 传播 | **DoS 成立** |
| VC-14 | NullReferenceException | 无 catch | **否** | 传播 | **DoS 成立** |
| VC-18 | OutOfMemoryException | 无 catch | **否** | 传播 | **DoS 成立** |
| VC-22 | IndexOutOfRangeException | 无 catch | **否** | 传播 | **DoS 成立** |
| VC-23 | FormatException | `catch(ClusterObjectNotFoundException)` :5607 | **否**（FormatException ≠ ClusterObjectNotFoundException） | 传播 | **DoS 成立** |

**标记为 False Positive 的 DoS**：
- **VC-05**: `catch(Exception)` at line 6648 捕获所有托管异常 → 不崩溃，仅 log
- **VC-06 的托管异常部分**: `catch(Exception)` at line 6599 捕获 → 不崩溃。**但 native AV 不可捕获**，仅当 OOB 跨越页边界时才触发 AV（概率低）

**OOB 读的重要补充**：
- VC-02/04/06/11/12 的 OOB 读大多发生在堆内存中，不跨页 → **不抛异常** → 无 DoS
- 不抛异常意味着程序继续用**垃圾数据**执行 → 可能导致逻辑错误或 InfoLeak，但不崩溃
- 只有极少数情况下 OOB 读跨越 4KB 页边界到未映射页才触发 AV → 不可捕获 → 进程终止

### VC-19 通知洪泛：DoS vs UAF/并发分析

#### 通知系统架构（基于代码）

**Producer（单线程）** — ClusApiAdapter.cs:7413-7502：
```
Worker.Start(() => {
    NotificationV2 notificationV = new NotificationV2();  // 单实例，复用
    while (!exitNotifications) {
        int ret = notificationV.GetNextNotification(notificationPort, 500);  // 500ms timeout
        // ... error handling ...
        NotificationArrived(notificationData.FilterAndType, ..., notificationData.BufferBuilder, notificationData.BufferSize);
    }
    isNotificationLoopStopped.Set();
});
```

**Buffer 生命周期** — NotificationDataV2.cs:
- `bufferBuilder = NativeMethods.Alloc(400)` — 构造时分配一次
- `GetNext()` 每次调用 `GetClusterNotifyV2(port, ..., bufferBuilder, ref bufferSize, ...)` — native API 写入同一块 buffer
- `ResetSizes()` 在 MoreData 时可能 `Free + Alloc` 重新分配
- `BufferBuilder` 属性直接返回原始 IntPtr

**Consumer（dispatcher 线程）** — NotificationManager：
- `DequeueNotification(500)` → 从 `notificationQueue` 取出 `Notification` 对象（托管对象，非 IntPtr）
- Producer 侧 `NotificationArrived` 在 handler 内部将数据提取为托管对象后 enqueue

#### 分析结论

| 检查项 | 结论 |
|-------|------|
| **是否存在并发 GetNotify?** | **否** — 单线程顺序循环，永远只有一个 outstanding GetClusterNotifyV2 |
| **Buffer 是否被异步使用?** | **否** — `NotificationArrived` 在 producer 线程内同步处理 buffer，处理完才进入下一次 GetNext |
| **Disconnect/Unsubscribe 竞态?** | **低风险** — `UnsubscribeNotifications` 设 `exitNotifications=true` 后 `WaitOne(30s)` 等待 loop 退出，然后才 dispose `notificationPort` |
| **30s 超时竞态?** | **理论存在**: 如果 handler 执行 >30s，`WaitOne` 超时 → `notificationPort.Dispose()` → producer 下次 `GetClusterNotifyV2` 用已 dispose 的 handle → InvalidHandle/AV。但 handler 通常 <100ms |
| **服务端能否控制 polling interval?** | **否** — 客户端硬编码 `timeout=500` (ms)。服务端只能控制通知的推送频率，不能让客户端更快轮询 |
| **洪泛效果?** | `GetClusterNotifyV2` 在有 pending 通知时立即返回（不等 500ms）→ producer 以最大速度处理 → **CPU 100% + notificationQueue 增长 → DoS** |

#### UAF 可利用性

**结论：UAF 概率极低**。原因：
1. 单 producer 线程，buffer 同步使用，无异步存储
2. Consumer 操作的是 `Notification` 托管对象，不持有 native IntPtr
3. Disconnect 有 30s wait（正常情况足够）
4. 唯一的 30s 超时竞态需要 handler >30s，但洪泛场景下 handler 执行快（几μs 的 ReadInt32），不可能超时

**VC-19 最终分级：DoS（CPU/内存耗尽），非 UAF**

### [VC-02] ReadMultiSzProp 无界指针递增 OOB Read

**A. 断言**
如果恶意服务端在 control code 响应的 property list 中，返回 MultiSz 属性的 `num2`（size 字段）> 实际 buffer 剩余大小，或数据中无 null terminator，则 [ClusApiAdapter.cs:317](Microsoft.FailoverClusters.Framework/MS.Internal.FailoverClusters.Framework/ClusApiAdapter.cs#L317) `Marshal.PtrToStringUni(buffer)` 越界读到 unmapped page → AccessViolation。

**缺失检查**: 无 `buffer + offset <= bufferEnd` 的边界校验；`PtrToStringUni` 无长度上限。

- **入口**: 调用返回含 MultiSz 属性的 control code（如 `CLUSCTL_RESOURCE_GET_PRIVATE_PROPERTIES` 对含 MultiSz 的资源类型）。

**但前置条件**: PropertyList 的值枚举循环 `for (*i; i += aligned(i[1]) + 8)` 依赖 endmark null DWORD 终止。如果 blob 合法结构完整（endmark 正确），字符串数据通常有 null terminator。攻击者需要构造一个 **endmark 正确但字符串内容无 null terminator** 的 blob — 这是可行的，因为 `cbLength` 可以包含非 null-terminated 字符串，ParseBuffer 只检查长度对齐不检查内容。

**E. 证据要求**
- [x] 代码路径可达
- [x] **已确认**: native 层使用 unbounded wcslen（PropertyList.c:515-519）
- [x] **已确认**: PropertyList 不复制 buffer，存储原始指针
- [ ] 需运行时 AV call stack（构造 cbLength=80 的字符串，内容全 'A' 无 null）

**F. 可利用性分级**: **DoS / Likely InfoLeak**（越界读；如果泄露数据通过后续 RPC 回传则为 InfoLeak）

**G. 最小修复建议**
1. `ReadMultiSzProp` 增加 `bufferSize` 参数，校验 `buffer + offset <= buffer + bufferSize`。

### [VC-03] ParseProperties 信任服务端 cbRoProperties/cbProperties

**A. 断言**
如果恶意服务端在 `CLUSTER_FILTER_GROUP_ITEM.cbRoProperties` 中传入 size > 实际 `pRoProperties` blob 大小，则 `CreatePropList(propertyList, propertyListSize)` at [ClusApiAdapter.cs:60](Microsoft.FailoverClusters.Framework/MS.Internal.FailoverClusters.Framework/ClusApiAdapter.cs#L60) 使 FrameworkSupport.dll 内部越界读。

**B. 攻击者可控面**
- `CLUSTER_FILTER_GROUP_ITEM` 结构体的 `cbRoProperties` / `cbProperties` 字段。
- 来源：`s_ApiCreateGroupEnum` / `s_ApiCreateEnumEx` 的 RPC 响应。
- IDL 中 property blob 使用 `[size_is]` 标记，**rpcrt4 按 `size_is` 分配 buffer**。

**C. 数据流与边界推导**
1. RPC 客户端 stub 按 `Member24` (cbRoProperties) 分配 `Member28` (pRoProperties) 的 buffer
2. 托管代码读取 `cbRoProperties` 并传给 `CreatePropList`
3. IDL: `[unique, size_is(Member24)] byte* Member28` — **rpcrt4 保证 Member28 指向 Member24 字节的已分配内存**
4. 因此 `CreatePropList(pRoProperties, cbRoProperties)` 的 size 与实际分配一致

**关键发现（已根据 native 逆向修正）**:

1. **RPC 层**: IDL `size_is` 保证 buffer 大小 = cbRoProperties。攻击者无法使 size > buffer。
2. **Native 层新发现**: PropertyList 的 `BuildListFromBuffer` **没有 ParseBuffer 式验证**（不像 ValueList 逐值校验 `offset + cbLength <= size`）。PropertyList 值枚举循环 `for (*i; i += aligned(i[1]) + 8)` 依赖 endmark null DWORD，**无边界检查**。
3. **但**: 因为 RPC `size_is` 保证 buffer=size，且 buffer 是 rpcrt4 分配的完整内存，攻击者能做的是在 blob 内放畸形结构数据。如果 `cbLength` 伪造极大但 endmark 缺失 → 值枚举循环读越界到 buffer 外。**PropertyList 不会像 ValueList 那样检测并抛异常。**
4. **Fuzzing 补充**: harness 使用随机数据 → endmark 大概率存在（4字节全0概率 ~2^-32/entry）→ 未触发越界。但**定向构造** "cbLength 极大 + 无 endmark" 的 blob **应该能触发**。

**D. 复现实验设计**
- 构造 blob: `[count=1][propName CLUSPROP_SZ: syntax=0x40003, cbLength=0x10000, data="A"*16][NO ENDMARK]`
- 传入 `CreatePropList(blob, sizeof(blob))` 其中 blob 实际只有 ~30 字节
- 观察: PropertyList 值枚举循环 `i += aligned(0x10000) + 8` → 指针跳过 64KB → AV

**E. 证据要求**
- [x] IDL `size_is` 保证 buffer 与 size 匹配
- [x] Fuzzing 无崩溃（但未定向测试 cbLength 伪造）
- [x] **已确认**: PropertyList 无 ParseBuffer 验证，值枚举循环无边界检查
- [ ] 需定向构造 blob 运行时验证

**F. 可利用性分级**: **DoS**（PropertyList 值枚举越界读 → AV；无写原语）

**G. 最小修复建议**
1. PropertyList 应增加类似 ValueList.ParseBuffer 的构造时验证。
2. 值枚举循环应检查 `(char*)i - bufferStart + 8 <= bufferSize`。

3. 需要 `4 + count * stride <= bufferSize`
4. 攻击者设 count=1000, bufferSize=100 → 第 (100-4)/stride+1 次迭代越界

**缺失检查**: 无 `count * stride + 4 <= bufferSize` 校验。

**D. 复现实验设计**
- **入口**: Mock server 实现 control code `CLUSCTL_RESOURCE_TYPE_REPLICATION_GET_REPLICATED_DISKS` 响应。
- **数据**: 返回 buffer = [count=100(2bytes), padding(2bytes), one_entry...], 但 bytesReturned = 4 + sizeof(one_entry)。
- **观测**: 第2次循环 `PtrToStructure` AV。开启 Page Heap 触发。
- **缩小变量**: VirtualAlloc guard page 在 buffer 末尾。

**E. 证据要求**
- [x] 代码路径可达（需 Replication 资源类型）
- [x] 边界条件明确
- [ ] 需运行时 AV call stack

**F. 可利用性分级**: **DoS**（越界读→AV；无写原语→不可RCE）

**G. 最小修复建议**
1. 添加 `if (4 + (long)num * offset > bufferSize) throw/return;`

### [VC-07] ExecuteOnControlCode — 二次 MoreData 不 realloc

**A. 断言**

**重新审视代码** (line 6308-6344):
```csharp
num = ClusterResourceControl(...); // 第二次
```

攻击者能否让第二次调用返回成功但 `bytesReturned > outBufferSize`? 这取决于 clusapi.dll 是否校验 `bytesReturned <= outBufferSize`。如果 clusapi 直接透传 RPC 响应中的 bytesReturned，则可以。

1. 第一次: MoreData → realloc to N → outBufferSize=N
2. 第二次: None, 但 bytesReturned=M (M>N, 由服务端控制)
3. `SafeCall(intPtr[N bytes allocated], M)` → 回调以 M 为 size 操作 N 大小的 buffer

**D. 复现实验设计**
- **入口**: Mock server 处理任意 control code。
- **数据**: 第一次返回 MoreData + bytesReturned=100; 第二次返回 Success + bytesReturned=10000。
- **观测**: 回调中 `Marshal.PtrToStructure` 或 `ParseProperties` 以 10000 bytes 大小操作 100 bytes buffer → Page Heap AV。

**E. 证据要求**
- [x] 代码逻辑明确
- [ ] 需确认 clusapi.dll 是否透传 bytesReturned（或内部校验）
- [ ] 需运行时 AV

**F. 可利用性分级**: **Memory Corruption (uncontrolled)**（服务端可控 size，读越界；写取决于回调行为）

**G. 最小修复建议**
1. 第二次调用后检查 `if (bytesReturned > outBufferSize) throw;`
2. 或改为循环 realloc 直到成功/超出上限。

### [VC-09] Icon2 整数溢出 → 堆损坏

**A. 断言**
如果服务端返回的 icon bitmap header 中 `biWidth * biBitCount` 溢出 int32，则 [Icon2.cs:439-440](Microsoft.FailoverClusters.Framework/Microsoft.FailoverClusters.Framework/Icon2.cs#L439-L440) 计算出小的 `num`，`new byte[num * (biHeight/2)]` 分配小数组，后续 `stream.Read(XOR, 0, XOR.Length)` 读入少量数据（不会溢出，因为 Read 按 Length 限制）。

**重新审视**: `stream.Read(base.XOR, 0, base.XOR.Length)` — Length 就是溢出后的小值。Stream.Read 最多读 Length 字节。所以**不会发生堆溢出写**。溢出只导致分配过小的数组，但 Read 按数组长度限制 → **数据截断**，不是堆损坏。

**但如果** `num` 溢出为负值 → `new byte[negative]` → OverflowException 或 OutOfMemoryException。

实际上 `(int)((biWidth * biBitCount + 31) & -32) >> 3` 在 unchecked 上下文中不会抛异常，但结果可能为负 → `new byte[负数]` → 转为 uint → 极大值 → OutOfMemoryException。

**B. 攻击者可控面**
- biWidth, biBitCount, biHeight 来自 icon stream（可嵌入集群资源属性中的 icon data）。

**C. 数据流与边界推导**
1. biWidth=0x10000, biBitCount=0x10000 → biWidth*biBitCount = 0x100000000 → 溢出为 0 (int32)
2. `(0 + 31) & -32 = 0` → `num = 0`
3. `new byte[0 * anything] = new byte[0]` → 空数组
4. `stream.Read(XOR, 0, 0)` → 读0字节 → 无崩溃
5. 其他组合可能产生负 num → 大数组 → OOM

**D. 复现实验设计**
- **入口**: 需要找到 Icon2 的调用者 — 可能是 snap-in 加载集群资源图标。
- **数据**: 畸形 BITMAPINFOHEADER 嵌入 icon stream。
- **观测**: OOM 或 OverflowException。

**E. 证据要求**
- [x] 整数溢出在代码中可见
- [ ] 需确认 icon data 来源是否为服务端可控
- [ ] stream.Read 按 Length 限制 → **不是堆溢出写**

**F. 可利用性分级**: **DoS**（OOM 或异常；不是 Memory Corruption — stream.Read 受 Length 限制）

Analysis_Result.md 声称 "Heap Corruption" 是**夸大的**。`stream.Read(buffer, 0, buffer.Length)` 不会写超过 buffer.Length。

**G. 最小修复建议**
1. 使用 `checked` 上下文包裹尺寸计算。

### [VC-16] 反射型信息泄露 (control code blob → string → RPC 回传)

**A. 断言**
如果 FrameworkSupport.dll 的 `GetStringProperty` 使用 blob 中的 size 字段（而非 wcslen）读取字符串，且 size > 实际字符串长度，则读入越界数据 → 编入 C# string → 后续 RPC 调用回传给服务端。

**B. 攻击者可控面**
- Control code 响应中的 raw byte property list blob（无 `[string]` 标记，按 `[size_is, length_is]` 传输）。
- blob 内部的 property value size 字段。

**C. 数据流**
1. `ClusterResourceControl` 返回 raw byte blob
2. `CreatePropList(blob, size)` → native 解析
3. `GetStringProperty` 返回字符串 → 如果越界读 → 含堆数据
4. 字符串被用于后续调用（如 `ResetCnoPassword(handle, dnsName)` at line 5378）
5. 后续 RPC 以 `[in, string]` 发送 → 服务端收到泄露数据

**Native 逆向已确认**: `PropertyData` 构造函数（PropertyList.c:515-519）使用 **unbounded wcslen**：
```c
v8 = v7 + 2;  // 指向 CLUSPROP_SZ.value_08（字符串数据）
v9 = -1;
do ++v9; while (v8[v9]);  // wcslen — 完全忽略 cbLength_04
std::wstring::assign(this + 24, v8, v9);  // 用 wcslen 结果作为长度
```
- `GetStringProperty` 通过 `ExecuteOnIterator` → `property_iterator::get_item` → `PropertyData::PropertyData` 提取字符串
- PropertyData **不使用 cbLength**，纯依赖 null terminator
- 如果 blob 中字符串无 null terminator（cbLength 填满非零字符），wcslen 越界读到后续堆数据
- 越界读到的数据被存入 `std::wstring` → 返回给托管层 → 可能通过后续 RPC 回传

**Fuzzing 为何未检测**: wcslen 越界读到合法堆内存（不跨页）时不会崩溃，fuzzing 仅检测 crash。InfoLeak 是静默的。

**D. 复现实验设计**
- **入口**: 直接 DllImport 调用 `CreatePropList` + `GetStringProperty`。
- **数据**: 构造 property list blob，string property 的 cbLength=80（40 wchar），数据全部为 'A'（无 null）。在 buffer 末尾之后紧跟已知 pattern（如 "SECRET"）。
- **方法**: VirtualAlloc 两页，第一页放 blob + pattern，第二页为 guard page。
- **观测**:
  - 如果 GetStringProperty 返回的字符串长度 > 40 → **InfoLeak 已证实**
  - 如果触发 AV（读到 guard page）→ **OOB Read 已证实**
  - 用 Windbg bp `PropertyData::PropertyData` 检查 v9（wcslen 结果）

**E. 证据要求**
- [x] **已确认**: native 使用 unbounded wcslen（PropertyList.c:515-519）
- [x] **已确认**: 不使用 cbLength 字段
- [ ] 需运行时证明返回字符串含越界数据（构造 harness）
- [ ] 需证明含泄露数据的字符串确实被 RPC 回传

**F. 可利用性分级**: **Manual Reflected InfoLeak**

已确认的反射回传路径：
- `collection["DnsName"].Value.ToString()` → `NativeMethods.ResetCnoPassword(handle, dnsName)` (line 5378-5379)
- 需用户手动触发"重置 CNO 密码"操作

升级到 **Proven Reflected InfoLeak** 需要：
1. Harness 证明 GetStringProperty 返回超长字符串（含越界堆数据）
2. 运行时抓包证明 ResetCnoPassword RPC 请求中 dnsName 参数包含越界数据

**G. 最小修复建议**
1. `PropertyData` 构造函数应使用 `cbLength` 限制 wcslen：`min(wcslen(v8), cbLength/2 - 1)`
2. 或在 `BuildListFromBuffer` 中增加 PropertyList 级别的 ParseBuffer 验证（检查每个字符串在 cbLength 范围内有 null terminator）

### [VC-24] Remote Extension DLL Loading → 管理员工作站 RCE

**A. 断言**

**D. 前提条件**
